\section{Implementación del núcleo funcional}

En esta segunda fase del proyecto nos centramos en pasar todas las ideas y la planificación realizada a un producto software. El objetivo principal ha sido construir una base que permitiera al usuario cubrir el flujo completo de la aplicación, desde el proceso de autenticarse hasta la gestión avanzada de las barajas y las colecciones.

Para asegurar la calidad de la aplicación, la implementación la realizamos siguiendo un enfoque modular e incremental. Este enfoque se basó en finalizar módulos concretos, lo cual nos permitió verificar de una forma constante todas las funcionalidades implementadas, antes de continuar con las siguientes, minimizando los riesgos con el producto.

\section{Desarrollo del backend y consolidación del CRUD}

En el backend, desarrollamos una API utilizando el entorno de .NET 8 diseñada para soportar todas las operaciones sobre las barajas de usuarios. Durante este desarrollo, se priorizo la separación de responsabilidades mediante el uso de clases DTO y de validaciones independientes, lo que nos permite mantener el código de una manera sencilla.

Sobre este sistema creamos un sistema CRUD completo para la entidad de barajas: creación (CREATE), lectura (READ), actualización (UPDATE) y borrado (DELETE). La lógica de actualización fue algo más difícil, ya que debíamos incluir la persistencia dinámica de las cartas dentro de cada uno de los mazos con sus respectivas reglas de dominio específicas, como son el caso de  las copias máximas permitidas o el tamaño total de la baraja.

Para garantizar que los datos externos que se utilizan fueran estables, mejoramos las peticiones con los proveedores de cartas mediante el desarrollo de un mejor cliente HTTP. Realizamos pequeños ajustes en los algoritmos de búsqueda para asegurar que las respuestas obtenidas desde las distintas fuentes fueran consistentes y estuvieran listas para ser usadas y procesadas.

La API desarrollada sigue un diseño REST, con recursos como usuarios, barajas y cartas. Esta organización nos ha permitido una separación clara de responsabilidades y nos facilita el poder escalar el sistema.

Los endpoints implementados cubren tanto la autenticación como la gestión completa de barajas y la colección de cartas del usuario.

\subsection*{Autenticación}

\begin{itemize}
    \item POST /api/auth/register: registro nuevos usuarios
    \item POST /api/auth/login: autenticación de usuarios existentes
\end{itemize}

\subsection*{Gestión de barajas (Decks)}

\begin{itemize}
    \item GET /api/decks: obtención de todas las barajas del usuario autenticado
    \item GET /api/decks/\{deckId\}: consulta detallada de baraja específica
    \item POST /api/decks: creación de nueva baraja
    \item PUT /api/decks/\{deckId\}: actualización de una baraja, incluyendo modificar cartas
    \item DELETE /api/decks/\{deckId\}: eliminación de una baraja
\end{itemize}

\subsection*{Gestión de colección de cartas (User Cards)}

\begin{itemize}
    \item GET api/user-cards: obtención de colección de cartas del usuario (README, detalles)
    \item GET api/user-cards/stats: obtención de estadísticas de colección (README, detalles)
    \item POST api/user-cards: inserción de nuevas cartas en colección
    \item POST api/user-cards/\{userCardId\}/quantity: ajuste incremental de copias carta
    \item DELETE api/user-cards/\{userCardId\}: eliminación carta de colección 
\end{itemize}

\subsection*{Búsqueda de cartas}

\begin{itemize}
    \item GET api/cards/search: búsqueda de cartas fuentes externas
\end{itemize}

Este conjunto de endpoints forma la base del sistema CRUD, no solo sobre barajas sino también sobre la colección personal del usuario. Los endpoints de estadísticas y búsquedas externas aportan un valor añadido al sistema. 

El funcionamiento detallado y su configuración, se encuentran explicados en el archivo README del repositorio, donde se tienen en cuenta aspectos como la lógica de interacción, las validaciones y la comunicación con la base de datos. 

\section{Desarrollo del frontend y flujo operativo real}

Mientras trabajabamos en el backend, también estuvimos desarrollando la interfaz de usuario que comenzó siendo un prototipo muy sencillo hasta llegar a un flujo completo y dinámico. Implementamos un sistema para la gestión de sesiones con las posibilidades de registro e inicio de sesión, garantizando que el usuario únicamente acceda a su colección almacenada en MongoDB una vez este se ha identificado.

Una vez completamos la validación de acceso, comenzamos a desarrollar las pantallas de colección y de detalle para cada barajas. Estas interfaces permiten a los usuarios interactuar en tiempo real con todo su contenido, editando su mazo y persistiendo con los cambios de forma inmediata con la API. Este avance en el proyecto supuso una refactorización del cliente por algunas áreas de responsabilidad para luego facilitar la escalabilidad futura.

Para finalizar el desarrollo de la interfaz, añadimos la funcionalidad de borrado definitivo de barajas y se añadieron diseños responsive a la aplicación. Estas mejoras ayudan a que la experiencia del usuario fuera la mejor posible.

\section{Capacidades multijuego y editor visual de baraja}

Una de las partes más importantes fue transformar la plataforma en un sistema capaz de integrar barajas procedentes de diferentes juegos. En la aplicación hemos integrado el tipo de juego ``gametype'' (Pokémon, Magic o Yu-Gi-Oh!) como un atributo dentro de la base de datos de MongoDB, permitiendo que el sistema se adapte a lo que el usuario seleccione cuando cree una nueva baraja.

Gracias a este diseño multijuego pudimos desarrollar un endpoint de búsqueda unificado capaz de consultar en múltiples fuentes externas. Con esta implementación permitimos que el usuario pueda buscar las cartas de cada una de las franquicias utilizando el mismo buscador, simplificando así la complejidad.

Para finalizar, construimos el editor de barajas. Esta herramienta permite que los usuarios añadan o retiren cartas mediantes controles directos, mientras el sistema se encarga de forma automática de las reglas de validación de cada uno de los juegos, garantizado que el mazo resultante sea legal dentro de cada uno de los juegos.

\begin{lstlisting}[language={[Sharp]C}, caption={Atributo GameType (MongoDB)}, label={cod:dev-multijuego-busqueda}, captionpos=b]
public static class DeckGameTypes
{
    public const string Pokemon = "pokemon";

var cards = normalizedGameType switch

\section{Resultados de la fase}

La fase 2 la terminamos de manera exitosa con un núcleo funcional completo que integra la autenticación, la vinculación de datos por usuarios, el CRUD completo y el motor multijuego. El sistema ha demostrado un correcto funcionamiento permitiéndonos la construcción de mazos completos de forma muy clara y sencilla.

Desde el punto de vista de la arquitectura, la aplicación queda preparada para añadir nuevos juegos de manera sencilla sin tener que reestructurar la base de datos para cada franquicia. 
De esta manera hemos terminado el desarrollo de toda la interfaz de la aplicación así como de la lógica de la misma, dejando así una buena base para el proyecto.
